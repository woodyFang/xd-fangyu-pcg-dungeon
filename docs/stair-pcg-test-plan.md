# 楼梯 PCG 独立验证方案

## 目标

用一张固定、可重复生成的测试塔，只验证楼梯空间契约，不把房间主题、装饰、战斗、随机图质量或普通走廊美术计入验收。

## 测试地图

- 4 层，每层一个大测试空间。
- F1 → F2：贴墙 L 型楼梯，验证结构主墙、开放侧护栏与转角内缝。
- F2 → F3：3 米宽直跑楼梯，验证 1 米地砖卡尺缩放和直跑楼板开洞。
- F3 → F4：开放区 L 型楼梯，验证没有实体墙时的完整护栏。
- 前两部楼梯固定位置和方向；第三部开放区楼梯使用确定性的合法候选选择，用来验证它不会让结构墙压住前一部楼梯的接入路线。

## 验收规则

1. 楼梯只连接相邻楼层。
2. 上下层共享完整楼梯井占位。
3. 楼板洞口只来自 `踏步标高 + 2.5m 净高` 与上层楼板的交集。
4. 上下落地区保持可行走，入口沿梯跑方向开放。
5. 上层落脚入口和梯段进入紧缩洞口的净空入口都必须无墙、无横向护栏；其余边界必须且只能归属实体墙或护栏。
6. 贴墙楼梯的结构侧使用实体墙，开放侧使用护栏。
7. L 型楼梯转角内缝不得生成横墙。
8. 5m 层高必须对应 20 级、每级 0.25m。
9. 三部楼梯组成的跨层链路必须通过最终三维连通验证。

## 后续改进建议

### 第一阶段：固定规则基线

- 保留当前 `stair contract` 作为唯一结构真相。
- 将测试塔加入每次楼梯修改后的定向回归。
- 失败信息按“候选、占位、洞口、墙护栏、连通”分类，避免只返回“无合法楼梯”。

### 第二阶段：候选评分可解释化

- 分开记录路线长度、结构改造、墙体贴合、共享房间重叠等奖惩。
- 在编辑器里显示候选被淘汰的具体原因。
- 自动候选优先选择能形成稳定结构墙、又不会扩大楼板洞口的方案。

### 第三阶段：结构参数化

- 将层高、净高、级高、最小梯跑长度和平台深度整理为一个楼梯规范对象。
- 由规范统一派生级数、踏步深度、楼板洞口和双层高墙，不允许各模块重复计算。
- 增加 U 型、双跑和电梯井时复用同一套占位、开洞与边界保护协议。

## 运行入口

- 浏览页面：`/stair-rule-test.html`
- 3D 操作：鼠标左键旋转、右键平移、滚轮缩放；可切换整体塔或单独聚焦三部楼梯。
- 3D 内容直接使用生成结果中的楼板洞口、双层高墙、梯级、平台、墙侧扶手和开放侧护栏数据。
- 自动测试：`node --test tests/stair-rule-map.test.js`
